# Merge Queue Skew Gate — Operator Runbook

**AISDLC-399 | Implemented 2026-05-23**

This runbook covers the `merge-queue-skew-gate` workflow — a GitHub Actions workflow that conditionally bypasses GitHub's `Update branches in merge queue` behavior when queued PRs are file-disjoint.

## Overview

GitHub merge queue's default `Update branches` setting forces a full rebase + CI re-run before each merge (~10-15 min per PR), even when queued PRs touch completely different files. This creates a serial throughput choke point: 5 queued PRs = 50-75 min of pure CI wait time.

The skew gate uses file-level overlap detection (the [Google TAP heuristic](https://abseil.io/resources/swe-book/html/ch23.html)) to determine whether a rebase is actually necessary:

- **Disjoint PRs** (no shared files): posts `merge-queue-skew-gate: success` immediately. GitHub merge queue proceeds without a full CI re-run cycle.
- **Overlapping PRs** (shared files): exits without posting the success status, allowing GitHub's default update-branch + CI re-run to run. Posts a comment naming the overlapping PRs for transparency.

**Prerequisite:** AISDLC-398 (content-addressed envelopes) — attestation envelopes now survive rebase without invalidation, so skipping update-branch on disjoint PRs does **not** cause v4-kick or re-sign loops.

---

## When the gate fires

The workflow fires on the `merge_group: checks_requested` event — every time a PR enters or is re-evaluated by the merge queue.

### Disjoint result (bypass activated)

```
Merge Queue Skew Gate — bypass activated
Result: Disjoint — no update-branch rebase needed.
Reason: No file overlap detected with other queued PRs.
```

The workflow posts `merge-queue-skew-gate: success` as a commit status check. GitHub merge queue treats this as a passing check and can proceed to merge without waiting for a full CI re-run.

**The PR comment includes:**
- The bypass reason (e.g. "only one PR in queue", "all other PRs touch disjoint files")
- A link to this runbook

### Overlap result (full update-branch required)

```
Merge Queue Skew Gate — full update-branch required
Result: File overlap detected with PR #<N>.
Overlapping files:
  src/some-shared-file.ts
  ...
```

The workflow exits without posting `merge-queue-skew-gate: success`, allowing GitHub's default update-branch + CI re-run to proceed. The PR comment names the overlapping PRs so the operator can understand why the full rebase cycle ran.

### Fail-safe behavior (AC-6)

If overlap detection fails (GitHub API error, git failure, malformed diff, or any unexpected error), the workflow defaults to **full update-branch** — it does NOT post the success status, so the conservative path always wins. A step summary note is added explaining the fail-safe activation.

## Excluded files (AC-4)

The following files are **never** counted as overlap, even if both PRs touch them. These are mechanical fixup files that don't represent real semantic ordering dependencies:

| Pattern | Reason |
|---|---|
| `.ai-sdlc/attestations/**` | Auto-generated per-commit DSSE envelopes |
| `pnpm-lock.yaml` | Lockfile regenerated by sibling `pnpm install` on rebase |
| `CHANGELOG.md` | Unreleased-section additions are always mergeable |
| `pipeline-cli/CHANGELOG.md` | Same as above |
| `orchestrator/CHANGELOG.md` | Same as above |
| `backlog/tasks/**` | Task lifecycle files, no code ordering dependency |
| `backlog/completed/**` | Closed task files, no code ordering dependency |
| `.ai-sdlc/verdicts/**` | Review verdict files, no code ordering dependency |

**If a PR touches ONLY excluded files**, it is treated as disjoint (no real overlap possible).

## How to override (force update-branch for a PR)

### Label override

Add the `force-update-branch` label to the PR before re-queuing. The skew gate detects this label and immediately returns the overlap result, bypassing file-overlap detection entirely.

```bash
gh pr edit <PR_NUMBER> --add-label "force-update-branch"
```

Remove the label after the PR merges (labels are not automatically cleaned up).

**Use when:**
- The PR has a semantic ordering dependency not captured by file-level overlap (e.g. interface + implementation in different packages)
- You want to guarantee a fresh CI run on the latest `main` regardless of overlap

### Emergency: disable the gate entirely

If the skew gate itself is causing problems, it can be bypassed by removing the `merge-queue-skew-gate` check from the merge queue's required status checks. The queue will fall back to its default update-branch behavior.

To restore, re-add `merge-queue-skew-gate` as a required check on the `main` ruleset in GitHub Settings → Rulesets.

## How to configure GitHub merge queue to respect the gate

For the skew gate to unlock queue bypass, the merge queue ruleset must list `merge-queue-skew-gate` as a **required status check on `merge_group` events**. Without this, GitHub doesn't know that a passing `merge-queue-skew-gate` means "skip update-branch".

**GitHub Settings → Rulesets → main → Required status checks:**
- Add `merge-queue-skew-gate` to the required checks list
- Ensure it applies to `merge_group` events (not just `pull_request`)

**Note on GH merge queue behavior:** As of 2026, GitHub does not expose a direct API to "skip update-branch for a specific PR". The skew gate achieves this effect by posting the status check *before* the update-branch step is triggered — if the required check is already passing when the PR enters the queue, the queue proceeds directly to merge. This is the same mechanism used by docs-only short-circuits in `ai-sdlc-gate.yml`.

## Reading the comment annotations

Every PR that passes through the merge queue will receive a comment from the skew gate. The comment structure:

### Bypass comment (disjoint)

```
Merge Queue Skew Gate — bypass activated

Result: Disjoint — no update-branch rebase needed.
Reason: <human-readable reason>

The merge queue can proceed without a full CI re-run cycle.
```

The `Reason` field is one of:
- `"This is the only PR in the merge queue."` — AC-5 (single PR, no overlap possible)
- `"This PR touches only mechanical fixup files (excluded from overlap detection)."` — AC-4 (this PR only modifies pnpm-lock.yaml, CHANGELOG, etc.)
- `"No file overlap detected with other queued PRs."` — general disjoint case

### Overlap comment (full update-branch)

```
Merge Queue Skew Gate — full update-branch required

Result: File overlap detected with PR #<N>.
Overlapping files:
  src/shared-file.ts

GitHub's default update-branch + CI re-run will proceed to prevent ordering skew.
```

The `Overlapping files` section lists the **exact files** that both PRs touch (after excluding mechanical files). This is the definitive signal that the two PRs have a real ordering dependency.

### Label override comment

When the `force-update-branch` label is present:

```
Merge Queue Skew Gate — full update-branch required

Result: Operator forced update-branch via 'force-update-branch' label on PR #<N>.
```

## Relationship to AISDLC-398

AISDLC-398 shipped content-addressed attestation envelopes — envelopes are now bound to the merge-base SHA rather than the exact file tree, so a `git rebase` that doesn't change any attested file content leaves the envelope valid.

This is what makes AISDLC-399 safe: previously, any rebase (even a trivial one triggered by update-branch on a disjoint PR) could invalidate the attestation envelope and trigger a v4-kick / re-sign loop. With content-addressed envelopes, even the overlap-driven update-branch path is safe — the re-sign only fires when the attested content actually changed.

**If AISDLC-398 is ever rolled back**, disable this skew gate first. Without content-addressed envelopes, bypassing update-branch could leave envelopes in an unverifiable state.

## Diagnostics

### Check gate status for a PR

```bash
# View the commit status check
gh pr checks <PR_NUMBER> | grep skew-gate

# View the PR comment
gh pr view <PR_NUMBER> --comments | grep -A 10 "Skew Gate"
```

### View gate run logs

```bash
# List recent workflow runs for the gate
gh run list --workflow merge-queue-skew-gate.yml --limit 10

# View a specific run's log
gh run view <RUN_ID> --log
```

### Force a re-run (if gate got stuck)

```bash
gh run rerun <RUN_ID> --failed
```
