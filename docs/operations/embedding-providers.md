## Configuration overview

The embedding framework is **off by default** in v1. Enable it by setting:

```bash
AI_SDLC_EMBEDDING_PROVIDER=1
```

When enabled, `Pipeline.spec.embedding` controls the provider and storage:

```yaml
spec:
  embedding:
    provider: openai-text-embedding-3-small   # adapter name
    storageBackend: jsonl                      # default; only option in v1
    staleVectorPolicy: lazy-re-embed           # 'lazy-re-embed' | 'fail-loud' | 'warn'
    gcRetentionDays: 90                        # per-org GC threshold
```

Per-org defaults live in `.ai-sdlc/embedding-config.yaml` (generated by `ai-sdlc init`).

### Required environment variables

| Adapter | Required env var |
|---|---|
| `openai-text-embedding-3-small` | `OPENAI_API_KEY` |
| Future adapters | Declared in adapter's `requires.envVar` |

### Pipeline-spec resolution (Phase 4 / AISDLC-340)

`Pipeline.spec.embedding` is OPTIONAL. When absent, the framework is disabled
and consumers emit `EmbeddingProviderNotConfigured`. When present AND
`AI_SDLC_EMBEDDING_PROVIDER=on`, pipeline-load:

1. Resolves `provider` against `orchestrator/src/embedding/registry.ts`
   (`UnknownEmbeddingProvider` on miss → pipeline-load fails fast).
2. Resolves `fallback` (if distinct) — fallback resolution failures also fail
   pipeline-load (preferable to silent fallback-unavailability at runtime).
3. Instantiates the named `storageBackend` via the factory at
   `orchestrator/src/embedding/storage/index.ts` (v1 ships `jsonl`).
4. Enforces the deprecation gate (see below).

Programmatic entry point: `loadEmbeddingFromPipelineSpec()` in
`orchestrator/src/embedding/pipeline-load.ts`.

## Choosing an adapter

| Adapter | Dimensions | Cost per 1M tokens | Best for |
|---|---|---|---|
| `openai-text-embedding-3-small` | 1536 | ~$0.02 | Default; broad coverage, cheap, mature. |
| `openai-text-embedding-3-large` | 3072 | ~$0.13 | When recall matters more than cost. |
| Cohere `embed-v3.0` (adopter-supplied) | 1024 | ~$0.10 | Multi-lingual; pay-per-token but different SLA. |
| Local ONNX / sentence-transformers (adopter-supplied) | 384–768 | $0 (compute only) | Air-gapped envs; embedding cost dominates pipeline cost. |

Selection criteria, ranked:

1. **Provenance constraints** — if you must NOT route source text to a 3rd-party API
   (regulated data, air-gapped), use a self-hosted adapter (`capabilities.selfHosted: true`).
2. **Consumer's stale-vector tolerance** — drift / trajectory consumers REQUIRE
   `fail-loud` (see next section). If your dominant consumer needs that, pick an adapter
   you do NOT anticipate deprecating in the next 6 months — a fail-loud consumer
   blocks ALL pipeline runs through a deprecation window. Pin a stable adapter.
3. **Cost shape** — `pay-per-token` (OpenAI, Cohere) is cheap at small corpora but
   compounds quickly; `subscription-quota` (future Anthropic embeddings) is flat-rate
   but routes through SubscriptionLedger. Use `cli-cost-report --unified` to
   measure actual spend before swapping.
4. **Dimension compatibility** — vectors from different adapters are NOT
   interchangeable. Swapping requires `cli-embedding-bump`.

## Stale-vector policy (per-org + per-consumer)

Per RFC-0019 OQ-2 re-walkthrough, stale-vector policy is configurable at TWO levels:

### Per-org default (`.ai-sdlc/embedding-config.yaml`)

```yaml
embedding:
  staleVectorPolicy:
    default: lazy-re-embed              # alternatives: fail-loud | warn
    perConsumerOverridesAllowed: true   # let consumers pin a stricter policy at the API site
```

| Policy | Behaviour | Best for |
|---|---|---|
| `lazy-re-embed` (default) | Silently re-embed stale vectors on read | PPA similarity, classifier embeddings, backlog auto-tagging — read-time consumers where freshness > history. |
| `fail-loud` | Refuse comparison + emit HIGH-severity Decision | Drift / trajectory consumers — RFC-0009 Eτ_tessellation_drift pins this at the API site. |
| `warn` | Log warning, return stale vector | NOT recommended — masks math errors in downstream consumers. |

### Per-consumer API-site override (the load-bearing pattern)

Consumers that genuinely cannot tolerate the org default pin their policy at the
API site, never inheriting org config. Concrete example in
`orchestrator/src/embedding/consumers/tessellation-drift.ts`:

```typescript
export const TESSELLATION_DRIFT_STALE_VECTOR_POLICY = 'fail-loud';

export async function embedDriftSignal(adapter: EmbeddingAdapter, text: string) {
  return adapter.embed(text, TESSELLATION_DRIFT_CONSUMER_LABEL);
}
```

**Why pin fail-loud for drift?** Drift reads historical trajectory across
successive document revisions. Lazy-re-embed silently overwrites old vectors,
destroying the time-series signal the rule is measuring. The historical
fidelity is load-bearing — if you ever surface a finding about how a soul drifted
over time, the underlying vectors MUST be the originals, not silently re-embedded
on yesterday's adapter version.

### Adding a new consumer

Mirror the `tessellation-drift.ts` pattern:

1. Export a `<NAME>_CONSUMER_LABEL` constant (cost-tracker dimension).
2. Export a `<NAME>_STALE_VECTOR_POLICY` constant (`'lazy-re-embed' | 'fail-loud' | 'warn'`).
3. Export a wrapper function that bakes both into every `adapter.embed()` call
   so the call site can't forget.
4. Add to the consumer registry section of this runbook so operators see the
   list of pinned policies at a glance.

## Deprecation lifecycle (with milestone dedup)

Per RFC-0019 OQ-4 re-walkthrough, deprecation uses **three-layer precedence**:

```
framework default (90d)
    ↓ overridden by
adapter-declared `capabilities.defaultGracePeriodDays`
    ↓ overridden by
per-org `spec.embedding.deprecationOverrides.gracePeriodDays`
```

Resolution is performed by `resolveEffectiveGracePeriodDays()` in
`orchestrator/src/embedding/pipeline-load.ts`. The effective value is what
shows up in the `DeprecationWarningEvent` operators see.

### Lifecycle phases

| Phase | Condition | Pipeline-load behaviour |
|---|---|---|
| Healthy | `now < deprecatedAt - gracePeriodDays` | No warning. |
| Warning window | `deprecatedAt - gracePeriodDays <= now < deprecatedAt` | Emit `DeprecationWarningEvent`. Pipeline continues. |
| Deprecated, default mode | `now >= deprecatedAt` AND `strictModeAtDeprecatedAt: false` | Emit warning event. Pipeline continues. |
| Deprecated, strict mode | `now >= deprecatedAt` AND `strictModeAtDeprecatedAt: true` | Throw `EmbeddingModelDeprecated`. Pipeline-load fails. |
| Removed | `now >= removedAt` | Throw `EmbeddingModelRemoved`. Pipeline-load ALWAYS fails. |

### Milestone dedup (catalog-flood prevention)

Under the autonomous orchestrator (RFC-0015), pipeline-load runs hundreds of
times per day. Per-load Decisions would flood the Decision Catalog.

The framework emits at milestone thresholds only:

```yaml
embedding:
  deprecation:
    catalogDedup:
      enabled: true
      milestonesDaysBeforeDeprecatedAt: [89, 60, 30, 7, 1]
```

The DEDUP itself is implemented by the orchestrator's Decision-writer (keyed
on `embedding-provider-deprecated:<adapter-name>:<deprecatedAt>`). The loader
emits one event per load; the writer deduplicates by Decision key + milestone.

### When you see a deprecation warning

1. Note the `replacementAlias` in the warning event.
2. Schedule a `cli-embedding-bump` run (next section).
3. Update `Pipeline.spec.embedding.provider` to the new adapter.
4. Re-deploy.

## Running embedding GC

The `cli-embedding-gc` tool removes stale vector store entries based on their
`writtenAt` timestamp. Run it periodically (e.g., weekly via cron) or manually
after large batch operations.

### Basic usage

```bash
# Run GC with default 90-day retention.
node pipeline-cli/bin/cli-embedding-gc.mjs run \
  --artifacts-dir .ai-sdlc/artifacts

# Per-org retention override (30 days):
node pipeline-cli/bin/cli-embedding-gc.mjs run \
  --artifacts-dir .ai-sdlc/artifacts \
  --retention-days 30

# Restrict to one provider:
node pipeline-cli/bin/cli-embedding-gc.mjs run \
  --artifacts-dir .ai-sdlc/artifacts \
  --provider openai-text-embedding-3-small
```

### Dry-run (preview without modifying files)

```bash
node pipeline-cli/bin/cli-embedding-gc.mjs run \
  --artifacts-dir .ai-sdlc/artifacts \
  --dry-run
# Output: [cli-embedding-gc] dry-run: would remove 142 of 1024 entries (older than 90 days)
```

### JSON output (for scripting)

```bash
node pipeline-cli/bin/cli-embedding-gc.mjs run \
  --artifacts-dir .ai-sdlc/artifacts \
  --format json
# Output: {"removed":142,"scanned":1024,"filesProcessed":2,"retentionDays":90}
```

### Retention boundary semantics

- Entries with `writtenAt` **older than** `retentionDays` days are removed.
- Entries written **exactly at** the boundary are retained (inclusive lower bound).
- Entries **without a `writtenAt` field** are treated as legacy/corrupt and retained
  (not silently dropped).

## Running `cli-embedding-bump`

> **Status:** Ships in Phase 3 (AISDLC-339). This section is a forward
> reference so the runbook reads correctly once Phase 3 lands. Treat the
> commands below as the contract Phase 3 must honor.

`cli-embedding-bump` is the migration tool operators run when:

- An adapter is approaching `deprecatedAt` and the deprecation warning surfaced.
- A `removedAt` is imminent (pipeline-load is about to fail).
- The operator deliberately switches `Pipeline.spec.embedding.provider` (e.g.,
  moving from `openai-text-embedding-3-small` to a larger model for recall).

### Dry-run (cost estimate, no writes)

```bash
node pipeline-cli/bin/cli-embedding-bump.mjs \
  --from openai-text-embedding-3-small \
  --to   openai-text-embedding-3-large \
  --artifacts-dir .ai-sdlc/artifacts \
  --dry-run
# Output: would re-embed 8,432 entries, est. cost $1.05 (3-large @ $0.13/1M tokens)
```

### Execute (atomic, keeps .bak)

```bash
node pipeline-cli/bin/cli-embedding-bump.mjs \
  --from openai-text-embedding-3-small \
  --to   openai-text-embedding-3-large \
  --artifacts-dir .ai-sdlc/artifacts \
  --execute
# Output: re-embedded 8,432 entries; original file preserved at <path>.bak
```

After verifying the new vectors integrate correctly (run your downstream
consumer's smoke test), remove the `.bak` file. Do NOT remove it until
you have verified — restoring from `.bak` is the recovery path.

### Concurrent-read safety

Mid-migration reads return either the OLD or NEW vectors (never a mix) per
the atomic-swap semantics of `cli-embedding-bump`. This is load-bearing for
pipelines that run continuously through the migration window — your drift
or similarity consumer won't observe inconsistent state.

## Inspecting the vector store

The `stats` subcommand provides a read-only view of the current store:

```bash
node pipeline-cli/bin/cli-embedding-gc.mjs stats \
  --artifacts-dir .ai-sdlc/artifacts

# Table output:
Provider                          ModelVersion  Count  OldestEntry  NewestEntry
--------------------------------  ------------  -----  -----------  -----------
openai-text-embedding-3-small     2024-01-25    1024   2026-01-01   2026-05-23
```

```bash
# JSON output for scripting:
node pipeline-cli/bin/cli-embedding-gc.mjs stats \
  --artifacts-dir .ai-sdlc/artifacts \
  --format json
```

The underlying JSONL files are human-readable and `jq`-friendly:

```bash
# Count entries by provider:
jq -r '.embeddingProvider' .ai-sdlc/artifacts/_embeddings/*.jsonl | sort | uniq -c

# Find entries written in the last 7 days:
jq 'select(.writtenAt > "2026-05-16")' .ai-sdlc/artifacts/_embeddings/*.jsonl

# Verify an entry's textHash:
jq 'select(.textHash == "<hash>")' .ai-sdlc/artifacts/_embeddings/*.jsonl
```

## Scale escalation

The JSONL backend is designed for corpora up to **~100K vectors per provider+version**.
When you cross either threshold:

- **> 100,000 entries** per `(provider, modelVersion)` tuple
- **p95 read latency > 250ms** for `read()` by textHash

...the backend emits an operator-visible signal (logged to stderr or your telemetry
callback) pointing to this runbook section.

### When to escalate

| Signal | Recommended action |
|---|---|
| Count > 100K | Evaluate sqlite backend (Phase 6+) or archive old entries via GC |
| p95 read > 250ms | Run GC to reduce corpus size; monitor; escalate to sqlite if persistent |
| Both thresholds crossed | Swap to sqlite backend immediately |

### Escalation path: JSONL → sqlite

> **Note:** sqlite backend is scheduled for Phase 6+ and is not yet available in v1.
> When it ships, the swap procedure will be:

1. Run a full GC pass to remove stale entries: `cli-embedding-gc run --retention-days 90`
2. Update `Pipeline.spec.embedding.storageBackend` to `'sqlite'`
3. Run `cli-embedding-bump --migrate-storage` to rewrite the JSONL files to sqlite
4. Verify the migration: `cli-embedding-gc stats` should show 0 JSONL entries
5. Monitor p95 read latency — sqlite indexed lookups should be <1ms for 1M entries

The framework's `EmbeddingStorageBackend` interface (at
`orchestrator/src/embedding/storage/types.ts`) is the extension point.
Adopter-supplied sqlite or pgvector backends implement the interface and are
wired into the factory at `orchestrator/src/embedding/storage/index.ts`.

### Escalation criteria (codified)

Per RFC-0019 OQ-1 re-walkthrough (2026-05-21):

> **Scale-escalation heuristic:** swap from JSONL to sqlite (or vector DB) when:
> - Count per `(provider, modelVersion)` exceeds ~100K entries, OR
> - p95 read latency exceeds 250ms.
> 
> Makes the transition operator-visible and corpus-driven, not tribal knowledge.

These thresholds are exported as:
- `SCALE_ESCALATION_MAX_ENTRIES = 100_000`
- `SCALE_ESCALATION_P95_READ_MS = 250`

from `orchestrator/src/embedding/storage/jsonl-backend.ts`.

## Unified cost report

Per RFC-0019 OQ-7 re-walkthrough, finance asks "what's our total AI spend
this month?" and needs ONE answer covering chat tokens, embedding tokens,
and subscription quota consumption. `cli-cost-report --unified` is that view.

### Basic invocation

```bash
node pipeline-cli/bin/cli-cost-report.mjs --unified \
  --cost-ledger-jsonl /tmp/cost-ledger-export.jsonl \
  --ledger-dir        .ai-sdlc/artifacts/_ledger \
  --since             2026-05-01T00:00:00Z \
  --format            text
```

Output (truncated):

------------------  -------------------  --------------------------------------  --------------------------------  ---------  ---------  -------

### Exporting the cost ledger

The orchestrator stores cost entries in SQLite by default. `cli-cost-report`
reads JSONL because pipeline-cli is orchestrator-free by design. Export the
ledger via a one-line node script:

```bash
node -e "
  const { Orchestrator } = require('@ai-sdlc/orchestrator');
  const orch = new Orchestrator({ statePath: '.ai-sdlc/state.db' });
  for (const e of orch.store.getCostEntries({})) {
    process.stdout.write(JSON.stringify(e) + '\n');
  }
  orch.close();
" > /tmp/cost-ledger-export.jsonl
```

The export is small (one line per entry) and trivially diffable with `jq`.

### Subscription cost-conversion math (caveats)

`subscription-quota` rows are computed from `SubscriptionLedger` window
state files. The conversion uses the simple model:

```
per-window-cost = monthlyUsd / (24 * 30 / windowHours)
window-cost     = per-window-cost * (consumedTokens / windowTokens)
```

Defaults (Claude Code Max-20x: `$200/mo`, 5h window, 200K tokens) are
overridable:

```bash
node pipeline-cli/bin/cli-cost-report.mjs --unified \
  --ledger-dir .ai-sdlc/artifacts/_ledger \
  --subscription-monthly-usd 50 \
  --subscription-window-hours 5 \
  --subscription-window-tokens 100000
```

The model is intentionally approximate — surfaces "subscription dollars at
play" without claiming line-item accuracy a subscription-billed substrate
cannot provide. For audited month-end reports, cross-check the
subscription-quota rows against the provider's invoice.

### Output formats

- `--format text` (default): operator-friendly table with TOTAL footer.
- `--format json`: array of `UnifiedCostRow` records; easy to pipe into
  `jq` for further analysis.
- `--format csv`: CSV with header `costModel,category,source,consumer,
  tokens,costUsd,recordCount`. Pipe into your data lake.

### Filtering

`--since <ISO>` filters the cost-ledger rows by `createdAt`. Subscription
ledger files are NOT filtered (they represent the current rolling window
state at the time of the read).

### Consumer attribution (OQ-6 re-walkthrough)

The `consumer` column reflects the `consumerLabel` parameter passed to
`adapter.embed()`. Cost-tracker records this dimension alongside
`(provider, modelVersion, accountId)` so the report can answer "drift cost
vs PPA cost" without re-instrumentation. Consumers that omit the label
appear as `unspecified` — this is a hint to the consumer's author that
they should add the label at the call site (mirror the `embedDriftSignal`
pattern in `orchestrator/src/embedding/consumers/tessellation-drift.ts`).

## Storage layout reference

```
<artifactsDir>/_embeddings/
├── openai-text-embedding-3-small-2024-01-25.jsonl  ← one file per (provider, modelVersion)
└── openai-text-embedding-3-large-2024-01-25.jsonl  ← (if multi-provider in use)
```

### Filename convention

Each file is named `<safeProvider>-<safeModelVersion>.jsonl` where each component is
sanitized to the regex `[a-zA-Z0-9._-]` (unsafe characters replaced with `-`). The
directory listing itself serves as the index — `scan()` walks `<embeddingsDir>/*.jsonl`
and each entry carries its own `(embeddingProvider, embeddingModelVersion)` provenance
for filtering. There is no separate index file (iter-2 review: dropping `_index.json`
eliminates the read-modify-write race on concurrent first-writes for different
provider/version tuples).

### JSONL entry format

Each line in a `.jsonl` file is a JSON object (the `VectorStoreEntry` schema,
`spec/schemas/vector-store-entry.v1.schema.json`):

```json
{
  "vector": [0.1, 0.2, 0.3, ...],
  "embeddingProvider": "openai-text-embedding-3-small",
  "embeddingModelVersion": "2024-01-25",
  "writtenAt": "2026-05-23T12:00:00.000Z",
  "text": "original source text",
  "textHash": "sha256hexhash",
  "metadata": { "sourceDoc": "rfc-0009.md" }
}
```

**Why `text` is stored:** Migration (`cli-embedding-bump`) requires re-embedding the
original text when a deprecated adapter is retired. Storing the vector without the
text makes migration impossible.

**Why `textHash` is stored:** Read-side deduplication uses the hash as the cache key
without recomputing it on every lookup.

**Provenance fields (`embeddingProvider`, `embeddingModelVersion`):** Both fields are
part of the vector's identity per RFC-0019 §2.3. Cross-provider comparisons are
mathematically invalid (different embedding spaces). The storage layout enforces
isolation by keeping one file per `(provider, modelVersion)` tuple.
